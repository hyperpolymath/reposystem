# Estate / Submodule Layout

How `reposystem` — the estate cockpit for the hyperpolymath org (estate #1) — physically lays out the repos it coordinates: a generated coordination manifest, eight consistent gitlink submodules, and a handful of directories embedded as ordinary trees by design. This page records the current, consolidated state (as of 2026-06-26) and the history that produced it.

The two governing rules, in one line each:

- **Flat clones are the source of truth for code.** `reposystem` owns no code history and pins nothing; it is a coordination layer. See `ESTATE-ORGANIZATION.adoc`.
- **`.gitmodules` must match `git ls-tree HEAD`.** Every declared submodule is now a `160000` gitlink; nothing is a "declared submodule committed as a plain tree" any more.

For where the schema and the visualisation sit on top of this layout, see [Architecture / Representation Model](Architecture-Representation-Model) and [Home](Home).

## 1. The manifests: `repos.toml` + `repos.groups.toml`

Two TOML files at the repo root index the estate. Neither pins code; they are coordination indices over the flat clones.

### `repos.toml` — generated coordination index

`repos.toml` is **generated, not hand-edited**. Its header is explicit:

```toml
# GENERATED by scripts/gen-repos-manifest.sh — DO NOT EDIT BY HAND.
# Regenerate: just repos-manifest
# Source of truth for code = flat clones in $REPOS_DIR (~/dev/repos).
# This file is only the aggregator's coordination index.
```

Its `[meta]` block declares the count:

```toml
[meta]
generated = "2026-05-18"
source = "repos-monorepo/.gitmodules"
count = 297
```

Each repo is a `[[repo]]` record with `name`, `path`, `url`, and a `kind` (e.g. `repo`, `julia`, `site`). The manifest importer (`src/importers/manifest.rs`) bridges this file into the `GraphStore` via `reposystem import manifest`. See [Architecture / Representation Model](Architecture-Representation-Model) for the import path.

### `repos.groups.toml` — hand-maintained epic index

`repos.groups.toml` is the complement: **hand-maintained, not generated**, and it survives `just repos-manifest`. Its header:

```toml
# repos.groups.toml — HAND-MAINTAINED. Not generated; survives `just repos-manifest`.
```

A "group" names the set of repos a cross-repo thread spans. The thread runner (`scripts/thread.sh`) resolves `--group <name>` to that list and then operates on the flat clones. You only list a group here when its membership is **not** derivable from a repo's `kind` (the runner also accepts e.g. `--kind julia` straight from `repos.toml`).

## 2. The eight gitlink submodules

`.gitmodules` declares exactly **eight** submodules, and `git ls-tree HEAD | grep '^160000'` returns all eight gitlinks — every declaration maps to a real `160000` gitlink, none broken or drifted. All are `git@github.com:hyperpolymath/<name>.git` remotes.

The one-line purposes below come from the tool registry (`config/tools.ncl`) where present; see [Tool Registry](Tool-Registry).

| Submodule | Registry role | Purpose |
|---|---|---|
| `git-scripts` | — | Estate git scripting helpers (gitlink only; not carried in `config/tools.ncl`). |
| `git-reticulator` | `utility` | Submodule / gitlink reticulation across the estate. |
| `rpa-elysium` | `utility` | Robotic process automation engine (workflows, scheduler, plugins). |
| `bitfuckit` | `utility` | Ada/SPARK repository auditor (health, compliance, security posture). |
| `contractiles` | `utility` | Declarative build/task contracts (ADJUST/INTENT/MUST/TRUST). |
| `recon-silly-ation` | `interop` | WASM document reconciliation (ReconForth VM). |
| `scaffoldia` | `utility` | Repository generator / customiser (trainyard topology). |
| `stateful-artefacts` | `interop` | Forge artefact hydration / consensus state (Paxos, DAX). |

In `config/tools.ncl` these seven (all but `git-scripts`) carry `status = 'extracted` — "now a real gitlink submodule (declared in `.gitmodules`)".

## 3. Embedded as a tree, by design

Four directories remain committed as ordinary `040000` trees rather than gitlinks. This is **intentional**, not drift:

| Path | Why it stays a tree |
|---|---|
| `total-upgrade` | An RSR scaffold kept embedded by design (the byte-identical duplicate `total-recall` was dropped — see history below). |
| `gui` | `reposystem`'s own web front-end (ReScript + Deno) — not a separable product. The 2026-06-02 verification records that `gui` declares no remote of its own and only references `reposystem`; "no standalone repo is expected and none exists." |
| `git-morph` | Carried in `config/tools.ncl` with `status = 'extracting` — "Still vendored as an embedded tree in reposystem; gitlink extraction pending." |
| `git-seo` | Same: `config/tools.ncl` marks it `status = 'extracting`, vendored pending extraction. |

So `git-morph` and `git-seo` are trees because their extraction is **pending**, while `total-upgrade` and `gui` are trees because they are **meant** to live inside the cockpit.

## 4. Consolidation history

The layout above is the endpoint of a deliberate, multi-PR consolidation. It began from a verified 2026-06-02 snapshot (in `docs/PHASE-2-STANDALONE-VS-LOCAL-VERIFICATION-2026-06-02.adoc`) where `.gitmodules` and the committed tree did **not** line up: orphan gitlinks with no URL, declared-but-vendored trees, and a byte-identical stub duplicate.

| PR | Change |
|---|---|
| **#130** | **Orphan gitlinks dropped** — `avatar-fabrication-facility` and `claim-forge` (each a gitlink with no `.gitmodules` URL **and** no forge repo) were removed (commit `f0586e8`/`741a982`, shipped with the ClusterFuzzLite Dockerfile / fuzz fix). After this, `git ls-tree HEAD` reports no orphan gitlinks. |
| **#131** | **`.gitmodules` reconciled** — the stale declarations whose paths were committed as ordinary trees rather than gitlinks had their `.gitmodules` sections removed (content kept). At this point `.gitmodules` declared only the two then-consistent gitlinks (`git-scripts`, `git-reticulator`). |
| **#133** | **`rpa-elysium` de-vendored + `total-recall` dropped** — the divergent 171-file `rpa-elysium` tree was replaced with a real gitlink at forge HEAD `bda6720`; `total-recall` (the byte-identical `total-upgrade` duplicate, tree `84d1361`) was dropped, with `total-upgrade` kept as an embedded RSR scaffold. |
| **#134** | **Five standalones extracted** — `bitfuckit`, `contractiles`, `recon-silly-ation`, `scaffoldia`, `stateful-artefacts` were created as GitHub forge repos and their vendored trees replaced with `160000` gitlinks. |

The displaced vendored content was **not** discarded. It is preserved on two pushed archive branches:

- `archive/rpa-elysium-vendored-2026-06-21` — the divergent `rpa-elysium` snapshot.
- `archive/vendored-standalones-2026-06-21` — the displaced standalones snapshot.

This consolidation was part of the same session that reduced hypatia neurosymbolic findings from 222 to 71 (the extraction removed vendored-tree scan noise). For the governance gate and the banned-language / anti-fiction rules that frame all of this, see [Governance and Hypatia](Governance-and-Hypatia).

## 5. How a submodule update works now

Because `.gitmodules` and `git ls-tree HEAD` agree on all eight gitlinks, ordinary git submodule mechanics work without choking on URL-less or tree-vs-gitlink mismatches.

```sh
# First clone, or pick up newly-added submodules:
git submodule update --init --recursive

# Verify the eight gitlinks are clean (no orphans, no drift):
git submodule status

# Confirm the committed objects are gitlinks (160000), not trees:
git ls-tree HEAD | grep '^160000'   # -> the eight submodules
```

`git-reticulator` is itself the estate's tool for "submodule / gitlink reticulation across the estate" (see [Tool Registry](Tool-Registry)); it is the intended mechanism for managing these gitlinks at estate scale rather than per-repo by hand.

> Note: the four by-design trees (`total-upgrade`, `gui`, `git-morph`, `git-seo`) are **not** submodules — they are part of `reposystem`'s own working tree and update with ordinary commits, not `git submodule update`.

== What Scaffoldia Is

Scaffoldia is the tool you use when you say *"I want to make a new repo."*

It is not a template cloner. It is an **orchestrator** that understands the
**topology of the hyperpolymath estate** вҖ” the ~300 repositories, the
services they run, the protocols they speak, and the junctions where they
connect. When you tell Scaffoldia what _kind_ of repo you want, it knows:

1. **What layers you need** вҖ” the structural components (Idris2 ABI, Zig FFI,
   Nickel manifests, Deno adapters, PanLL panels, etc.)
2. **Where the repo sits in the network** вҖ” which estate systems it wires
   into (BoJ catalogue, Groove mesh, VeriSimDB, launcher-scaffolder, etc.)
3. **What to ask you** вҖ” the configuration questions specific to this kind
   (port number, capability domain, target languages, social image, etc.)
4. **How to wire it in** вҖ” the ordered post-mint provisioning sequence
   (GitHub creation, rulesets, mirroring, catalogue registration, etc.)

== The Trainyard Model

The hyperpolymath estate is a **trainyard**. Each repository sits on a
**track** (its architectural category), and connects to other tracks at
**junctions** (integration points with shared services).

== The Four Roles

Scaffoldia has four roles, each handling a different phase of repo creation:

=== 1. Minter

*"Here are the files you need."*

The minter generates the file structure for a repo kind. It uses templates
with variable substitution (Nickel for config, `{{var}}` for text) and
knows which layers each kind requires.

A BoJ cartridge mints: `abi/`, `ffi/`, `adapter/`, `panels/`, `cartridge.ncl`,
`mod.js`, `README.adoc`, `EXPLAINME.adoc`, etc.

A Rust CLI mints: `Cargo.toml`, `src/main.rs`, launcher script, etc.

=== 2. Provisioner

*"Now let me wire it into the estate."*

After minting, the provisioner runs the ordered sequence of post-mint
actions: create the GitHub repo, apply rulesets, set up mirroring, register
in catalogues, provision databases, wire protocols, generate launchers.

Some steps are automatic (shell commands). Others are interactive (need
human decisions). The provisioner knows which is which and pauses at
interactive steps.

=== 3. Configurator

*"Let me ask you the right questions."*

Each repo kind has a set of minting prompts вҖ” the questions Scaffoldia asks
during creation. A BoJ cartridge asks for domain, port, and protocols. A
Tauri app asks about mobile targets. A game asks whether it's co-developed
(AGPL) or solo (PMPL).

The configurator validates answers against estate constraints (e.g. port
numbers against TOPOLOGY.md, repo names against existing repos).

=== 4. Harness

*"Here's your test and CI scaffolding."*

The harness generates the test infrastructure matched to the repo kind:
Zig tests for FFI code, Idris2 proofs for ABI code, ReScript tests for
TEA modules, and the 17 standard RSR GitHub Actions workflows.

== Relationship to Other Tools

[cols="1,2"]
|===
| Tool | Relationship

| **rsr-template-repo**
| The RSR base template. Scaffoldia uses it as the foundation, then adds
  kind-specific layers on top. `just init` from the template is step zero.

| **launch-scaffolder**
| Generates launchers (.sh + .desktop). Scaffoldia delegates to it during
  provisioning for repo kinds that need launchers (Tauri, Dioxus, Rust CLI).

| **proven-servers**
| Formally verified server protocols. Scaffoldia wires new ProvenServer
  repos into the proven-agentic protocol mesh.

| **boj-server**
| The BoJ cartridge registry. Scaffoldia registers new BojCartridge repos
  in Catalogue.idr and TOPOLOGY.md.

| **standards**
| The A2ML/RSR specification repo. StandardsExtension repos extend it.
|===
